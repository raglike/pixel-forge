# PixelForge 像素动画工具 - 知识图谱

## 一、像素动画基础知识

### 1.1 像素艺术原理

像素艺术（Pixel Art）是一种以离散的像素点为基本单位构建图像的数字艺术形式，源于早期低分辨率显示设备（8-bit/16-bit时代），现已成为一种独立的艺术风格。

#### 核心参数

| 参数 | 说明 | 对游戏/动画的影响 |
|------|------|-------------------|
| **分辨率 (Resolution)** | 图像的像素宽×高，如 32×32、64×64 | 决定细节丰富度，文件大小与分辨率平方成正比 |
| **调色板 (Palette)** | 可使用的颜色集合，通常 2-256 色 | 限制颜色数量影响文件体积和复古感，调色板管理是像素艺术核心技能 |
| **DPI (Dots Per Inch)** | 打印分辨率，与像素艺术关系不大 | 屏幕显示通常72-96 DPI，导出打印才需考虑 |

#### 像素艺术分类

- **1-bit**: 仅黑白两色（如《Monument Valley》的极简风格）
- **2-bit**: 4色灰度
- **8-bit**: 256色（SNES/Genesis时代标准）
- **16-bit**: 65536色（PS1/N64时代）
- **32-bit / True Color**: 1677万色（现代"高清像素"风格）

### 1.2 常见像素尺寸

像素艺术的尺寸选择取决于用途：

| 尺寸 | 像素总数 | 典型用途 |
|------|----------|----------|
| **8×8** | 64 | 图标、粒子、最小角色（如《Minecraft》生物） |
| **16×16** | 256 | 小型道具、Tile地图块、游戏角色（如早期《宝可梦》角色） |
| **32×32** | 1,024 | 中型角色、较大道具（《Celeste》角色约32px高） |
| **64×64** | 4,096 | 大型Sprite、精细角色（《Hyper Light Drifter》主角约48-64px） |
| **128×128** | 16,384 | 精细像素艺术、复杂角色、UI背景 |
| **256×256** | 65,536 | 超大Sprite、高清像素艺术展示帧 |

#### 尺寸选择建议

```
角色高度参考（像素）：
├─ 8px:   超微型单位（粒子、小生物）
├─ 16px:  小型敌人、道具
├─ 32px:  主角（《Celeste》）
├─ 48px:  中型角色（《Hyper Light Drifter》）
└─ 64px+: 大型Boss、精细角色艺术
```

**重要原则**：像素艺术尺寸通常为 8 的倍数，便于Tile地图拼接。

### 1.3 动画帧率

像素动画帧率决定动画流畅度和文件大小：

| 帧率 | 适用场景 | 帧间隔 | 备注 |
|------|----------|--------|------|
| **8 fps** | 复古风格、极简动画、特效序列 | 125ms | 典型的复古游戏感（早期FC游戏） |
| **12 fps** | 2D游戏常见帧率、步行动画 | 83ms | 平衡流畅度与文件大小 |
| **15 fps** | 标准像素动画 | 67ms | 《宝可梦》水晶/红宝石帧率 |
| **24 fps** | 电影级动画下限 | 42ms | 与电影帧率对齐 |
| **30 fps** | 现代2D游戏常用 | 33ms | 流畅但文件较大 |

#### 帧率与动画类型匹配

- **待机/呼吸动画**: 4-8 fps（微妙摆动）
- **移动/走路循环**: 8-12 fps（4-8帧构成完整循环）
- **攻击/动作动画**: 12-24 fps（快速动作需要更高帧率）
- **特效/粒子**: 30+ fps 或可变帧率

## 二、技术实现方案

### 2.1 图片像素化算法

将高清图片转换为像素风格是工具的核心功能。

#### 2.1.1 像素化算法对比

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **最近邻插值 (Nearest Neighbor)** | 每个目标像素取源图最近像素值 | 保持硬边缘，无插值模糊，**最保真像素感** | 锯齿严重，缩小失真大 | 缩小到低分辨率（最佳像素艺术效果） |
| **双线性插值 (Bilinear)** | 4个最近像素加权平均 | 平滑过渡，缩小效果好 | **模糊像素边缘**，失去锐利感 | 不推荐用于像素艺术 |
| **双三次插值 (Bicubic)** | 16个像素加权，更平滑 | 缩小效果好 | **完全破坏像素感** | 预处理参考，不用 |
| **Lanczos重采样** | sinc函数加权 | 高质量缩小 | **丢失像素艺术特征** | 不适用 |
| **自定义像素化** | 区域采样+调色板映射 | 可精确控制效果 | 需要更多参数调优 | **推荐方案** |

#### 2.1.2 推荐算法：改进的像素化流程

```
原始图片 → 缩小到目标分辨率（最近邻） → 调色板量化 → 输出像素图
```

**关键步骤**：
1. 使用**最近邻插值**将图片缩放到目标像素尺寸
2. 对缩小后的图片进行**调色板量化**（颜色数量压缩）
3. 可选：**边缘锐化**处理恢复清晰度

#### 2.1.3 棋盘格渲染（Pixel-perfect Rendering）

在高清屏幕上正确显示像素艺术的技术：
- 检测显示设备的 pixel ratio
- 使用 `image-rendering: pixelated` CSS 属性
- 或在Canvas中使用 `imageSmoothingEnabled = false`

### 2.2 调色板生成算法

调色板决定像素艺术的颜色风格。常用算法：

#### 2.2.1 算法对比

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **中位切分 (Median Cut)** | 在颜色空间中递归二分颜色桶 | 质量高，忠实原图颜色分布 | 计算较慢 |
| **八叉树量化 (Octree)** | 用八叉树组织颜色，按重要性剪枝 | **速度快**，支持渐进式 | 颜色边界可能不平滑 |
| **K-Means聚类** | 将颜色聚类到K个中心 | 可精确控制颜色数量 | 需要迭代，计算量大 |
| **自适应调色板** | 根据图片颜色分布动态生成 | 最佳颜色利用率 | 实现复杂 |

#### 2.2.2 固定调色板方案

针对像素艺术常用的固定调色板：
- **PICO-8**: 16色固定调色板（经典糖果色）
- **NES**: 54色可用调色板
- **GameBoy**: 4色灰度（#0f380f, #306230, #8bac0f, #9bbc0f）
- ** Commodore 64**: 16色调色板

#### 2.2.3 主色调检测

检测图片主色调的方法：
1. **K-Means聚类**：找出N个主要颜色
2. **颜色直方图**：统计颜色频率，取Top N
3. **颜色空间分析**：在LAB色彩空间计算颜色距离

### 2.3 前端实现方案

#### 2.3.1 Canvas 2D API

**推荐用于**：大多数像素动画工具场景

```javascript
// 像素化处理核心代码
const canvas = document.createElement('canvas');
const ctx = canvas.getContext('2d');
ctx.imageSmoothingEnabled = false; // 关键：禁用平滑

// 绘制并缩放
ctx.drawImage(sourceImage, 0, 0, targetWidth, targetHeight);

// 获取像素数据
const imageData = ctx.getImageData(0, 0, targetWidth, targetHeight);
```

**优点**：API简单，兼容性好，像素操作灵活
**缺点**：大量帧处理时性能有限

#### 2.3.2 WebGL / WebGL2

**推荐用于**：需要实时滤镜、性能要求高的场景

- 可使用 **着色器 (Shader)** 进行GPU加速像素化
- 支持更多图像混合模式和滤镜效果
- 学习曲线较陡

#### 2.3.3 方案选型建议

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 基础像素化工具 | **Canvas 2D** | 实现简单，够用 |
| 实时预览/滤镜 | **WebGL** | GPU加速，流畅 |
| 导出处理（后台） | **OffscreenCanvas + Web Worker** | 不阻塞UI |

### 2.4 动画导出方案

#### 2.4.1 GIF 导出

- **优点**：广泛兼容，所有浏览器支持，无需插件
- **缺点**：最大256色，有透明抖动问题，文件较大

#### 2.4.2 序列帧导出

- **优点**：无损，可导出任意格式（PNG序列）
- **缺点**：文件数量多，需要打包使用

#### 2.4.3 Sprite Sheet 导出

- **优点**：一次HTTP请求，GPU友好的图集布局
- **缺点**：需要配套的JSON metadata文件

## 三、导出格式详解

### 3.1 GIF 格式

#### 格式原理

GIF (Graphics Interchange Format) 使用 **LZW无损压缩**，但受限于：
- **最大256色调色板**（8-bit）
- **单帧延迟时间**（centiseconds，1/100秒）
- **透明色**：1-bit透明（要么完全透明，要么不透明）

#### GIF 动画结构

```
GIF Header
├─ Logical Screen Descriptor (尺寸、调色板)
├─ Global Color Table (256色)
└─ Netscape Application Extension (循环次数)

Data Sub-blocks:
├─ Graphic Control Extension (延迟、透明度)
├─ Image Descriptor (帧位置、尺寸)
├─ Local Color Table (可选，本地调色板)
└─ Image Data (LZW压缩像素数据)

GIF Trailer (0x3B)
```

#### 延迟时间计算

```javascript
// 12fps = 83.33ms ≈ 83 centiseconds
const delayCentiseconds = Math.round(1000 / fps);
// 注意：有些工具使用 ceil，有些使用 floor，需实测
```

#### GIF 限制与解决方案

| 限制 | 问题 | 解决方案 |
|------|------|----------|
| 256色 | 复杂动画颜色不足 | 使用**颜色抖动(Dithering)** |
| 锯齿 | 缩放后边缘不平滑 | **抖动+调色板优化** |
| 文件大 | 多帧动画体积大 | **帧差异传输**（只存储变化区域） |

### 3.3 Aseprite 格式

#### .ase (Aseprite Native Format)

- **二进制格式**，支持图层、帧、调色板
- 支持 **Cel (Animation Cel)** 层级系统
- 支持帧标签（Frame Tags）标记动画区间

#### Aseprite JSON 导出格式

```json
{
  "frames": [
    {
      "filename": "frame_0000.png",
      "frame": {"x": 0, "y": 0, "w": 32, "h": 32},
      "duration": 80
    }
  ],
  "meta": {
    "app": "Aseprite",
    "version": "1.3",
    "image": "export.png",
    "size": {"w": 128, "h": 32},
    "frameTags": [
      {"name": "walk", "from": 0, "to": 3, "direction": "forward"}
    ]
  }
}
```

### 3.4 游戏引擎导入方式

#### Unity Sprite 导入

1. 将Sprite Sheet图片导入Unity
2. **Sprite Mode**: 设置为 `Multiple`（多帧）
3. **Sprite Editor**: 打开编辑器，手动或自动切割帧
4. **Pixels Per Unit**: 设置为 sprite 的像素高度（如64）
5. 生成 **Sprite Atlas** 优化运行时性能

```csharp
// 运行时播放Sprite动画
SpriteRenderer sr;
Sprite[] frames;
float frameRate = 12f;

void Update() {
    int index = (int)(Time.time * frameRate) % frames.Length;
    sr.sprite = frames[index];
}
```

#### Godot SpriteSheet 导入

1. 导入图片，设置 `Texture import > No Filter`（像素完美）
2. 使用 **SpriteFrames** 资源创建动画
3. 或使用 **AnimatedSprite2D** + `SpriteFrames` 自动播放
4. 可直接导入Aseprite `.ase` 文件（Godot 4.x支持）

#### Unreal Engine Pixel Sprite

- 使用 **Paper2D** 插件
- **Flipbook** 组件播放序列帧动画
- 支持Sprite Sheet切割

## 四、竞品分析

### 4.1 像素编辑器对比

| 工具 | 类型 | 平台 | 价格 | 特色功能 |
|------|------|------|------|----------|
| **Aseprite** | 专业像素艺术 | Win/Mac/Linux | $20 (买断) | 图层、动画时间轴、调色板管理、脚本API |
| **Piskel** | 在线像素编辑器 | Web | 免费 | 协作、导出GIF/PNG序列 |
| **Leshy SpriteMaker** | 工具/生成器 | Web/Mac | 免费/$6 | Sprite Sheet切割、预设生成 |

| **Pixelmator Pro** | 位图编辑 | Mac | $120 | 机器学习增强，非专精像素 |

### 4.2 Aseprite 详细分析

**优势**：
- 完整的图层系统（背景层、动画层）
- 专业的动画时间轴（ Onion Skinning 洋葱皮）
- 内置调色板管理器和颜色选择器
- 支持动画帧标签（walk、idle、attack）
- **脚本API**（Lua）可扩展
- 导出为GIF、PNG序列、Sprite Sheet、.ase

**局限性**：
- 无AI辅助生成功能
- 无在线协作
- 界面较专业，有学习曲线

### 4.3 Piskel 详细分析

**优势**：
- 完全基于浏览器，无需安装
- 基础像素编辑功能完整
- 免费开源（GitHub）
- 可导出GIF动画

**局限性**：
- 无图层系统
- 无高级动画工具（洋葱皮等）
- 依赖网络，大文件性能差

### 4.4 Leshy SpriteMaker 详细分析

**定位**：Sprite Sheet处理工具，非完整编辑器

**功能**：
- 导入Sprite Sheet自动切割
- 预设模板（2×2、3×3、4×4等）
- 颜色提取和调色板生成

## 五、开源工具/库

### 5.1 GIF 生成库

#### 5.1.1 gif.js

- **类型**：前端库（Web Worker加速）
- **特点**：多帧GIF编码，支持质量配置
- **限制**：需要配置worker路径

```javascript
import GIF from 'gif.js';

const gif = new GIF({
  workers: 2,
  quality: 10, // 1-20，数字越小质量越好
  width: 128,
  height: 128,
  workerScript: '/gif.worker.js'
});

frames.forEach(frame => gif.addFrame(frame, { delay: 83 }));
gif.on('finished', blob => saveAs(blob, 'animation.gif'));
gif.render();
```

#### 5.1.2 gifshot

- **类型**：纯JS实现（无Worker）
- **特点**：轻量级，简单API

#### 5.1.3 omggif

- **类型**：纯JS，Gzip压缩后仅2KB
- **特点**：极小体积，适合嵌入式

#### 5.1.4 库选型对比

| 库 | 体积 | Worker支持 | 质量 | 备注 |
|----|------|------------|------|------|
| **gif.js** | ~50KB+worker | ✅ | 中等 | 需要额外配置，成熟方案 |
| **gifshot** | ~30KB | ❌ | 中等 | 简单场景够用 |
| **omggif** | ~2KB | ❌ | 基础 | 极简需求首选 |

### 5.2 游戏框架

#### Phaser.js

- **类型**：2D游戏框架
- **Sprite Sheet支持**：原生支持，自动切割
- **动画系统**：`Phaser.Animations.Animation`

## 六、游戏引擎兼容

### 6.1 Unity 规范

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **Pixels Per Unit (PPU)** | Sprite高度 | 64px角色 → PPU=64 |
| **Filter Mode** | **Point (no filter)** | 保证像素完美 |
| **Compression** | None 或 Low | 像素艺术避免压缩失真 |
| **Max Size** | ≥ 图片实际尺寸 | 避免强制缩放 |

### 6.2 Godot 规范

| 参数 | 设置 | 说明 |
|------|------|------|
| **Import → Filter** | **Nearest** | 禁用线性过滤 |
| **压缩** | 无损/VRAM压缩关 | 保护像素质量 |

### 6.3 Unreal Paper2D

- 需要启用 **Paper2D** 插件
- **Sprite** 资源类型存储单帧
- **Flipbook** 播放序列帧动画

## 七、技术选型建议

### 7.1 前端框架

| 方案 | 适用场景 | 推荐度 |
|------|----------|--------|
| **React + TypeScript** | 中大型项目，组件化需求高 | ⭐⭐⭐⭐⭐ |
| **Vue 3 + TypeScript** | 快速原型，中小型项目 | ⭐⭐⭐⭐ |
| **原生 HTML/JS** | 极简工具，无需框架 | ⭐⭐⭐ |

**推荐**：React + TypeScript，原因：
- 组件化适合编辑器的复杂UI状态管理
- 生态丰富（gif.js等库有React绑定）
- TypeScript类型安全

### 7.2 图像处理库

| 功能 | 推荐库 | 理由 |
|------|--------|------|
| **Canvas像素操作** | 原生Canvas API | 足够，无需额外库 |
| **GIF编码** | **gif.js** | Worker支持，质量可控 |
| **GIF解码** | **omggif** | 轻量，导入现有GIF |
| **Sprite Sheet处理** | **TexturePacker.js** 或自研 | 自研更灵活 |

### 7.3 导出方案组合

| 目标 | 推荐格式 | 配套 |
|------|----------|------|
| **社交分享/预览** | GIF | gif.js，质量10 |
| **游戏开发** | Sprite Sheet + JSON | 自定义格式或Phaser Hash |
| **专业像素艺术家** | PNG序列 + Aseprite JSON | Aseprite兼容 |
| **Unity开发** | Sprite Sheet + JSON | Unity可解析的metadata |

### 7.4 推荐技术栈

```
核心层:
```
